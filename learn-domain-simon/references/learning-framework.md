# Simon-Method Learning Framework

Use this reference to shape the learning experience. Keep the output practical, compact, and suitable for repeated weekly use.

## Core Principles

- Use goal-first learning: define what the learner wants to be able to do, not only what they want to know.
- Fit the plan to the learner profile: current level, target use case, weekly time budget, and preferred theory/practice balance.
- Build a knowledge map before diving into details so the learner can see the domain structure.
- Make the knowledge map dependency-aware: show prerequisite concepts, core concepts, advanced branches, and recommended learning order.
- Decompose the domain into small chunks that can be learned and tested independently.
- Teach each chunk with a short explanation, then immediately force retrieval through questions.
- Add deliverable-based practice after questioning so knowledge becomes usable and inspectable.
- Compress weekly and monthly learning into one-page summaries to form larger reusable chunks.

## Plan Shape

For each domain, create this progression:

1. Knowledge map: concepts, relationships, dependencies, and common paths through the field.
2. Micro-chunks: concepts small enough to learn in one focused session.
3. Teaching: explain what the chunk is, why it matters, and how it connects to earlier chunks.
4. Questioning: ask layered questions that reveal whether the learner can recall, explain, transfer, and identify counterexamples.
5. Practice: require a concrete deliverable such as a notebook, demo, case analysis, code repository, decision memo, reading card, diagram, calculation, or mini-project.
6. Compression: turn many chunks into one-page summaries and higher-order mental models.
7. Synthesis project: combine the most important chunks into one final project with explicit evaluation criteria.

## Chunk Design

Each chunk should have:

- A short name.
- One central concept.
- A clear dependency, if any.
- A teaching explanation.
- Layered questions:
  - Recall: definitions, facts, steps, and distinctions.
  - Explanation: why it works and how it connects to nearby concepts.
  - Transfer: how to apply it in a new case.
  - Counterexample: when it fails, does not apply, or misleads.
- One deliverable-based practice task.
- A completion standard tied to both understanding and artifact quality.

Avoid making chunks so large that they require multiple unrelated mental models.

## Weekly One-Page Compression

Use these fields:

- This week's core concepts.
- The five-sentence explanation.
- One diagram or concept tree.
- Three examples and one counterexample.
- Mistakes, confusions, and corrected understanding.
- Questions to carry forward.
- One deliverable produced this week.
- Next week's highest-leverage chunk.

## Monthly Chunk Packaging

Use these fields:

- Chunks learned this month.
- Larger modules formed from those chunks.
- Reusable models or frameworks.
- What can now be done that could not be done before.
- Remaining weak links.
- One transfer task for a different context.
- Updated knowledge map.
- Final or monthly synthesis deliverable.

## Final Project Design

When the domain supports practice, add one synthesis project:

- Combine three to six core chunks.
- Require a visible deliverable.
- Include evaluation criteria.
- Add one easier version and one stretch version.
- Prefer realistic artifacts over abstract homework.

## Handling Broad Domains

If the field is broad, ask the user to choose one of these narrowing lenses before planning:

- Career or role.
- Project outcome.
- Technical subfield.
- Business scenario.
- Exam or certification.
- Beginner literacy versus practical mastery.
